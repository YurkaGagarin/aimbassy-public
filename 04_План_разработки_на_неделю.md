# План разработки на неделю (соло)

> Закрывает: «план, с каких узлов начать (критичных), какие отложить, чтобы успеть за неделю с Claude Code Opus 4.8».
> Принцип: сначала оцениваемое ядро (Кольцо A), потом витрина (B), мечта (C) — только на бумаге.

## Золотое правило недели
**Каждый день должен заканчиваться чем-то, что работает end-to-end.** Лучше «кривой, но целый» бот в день 2, который ты потом улучшаешь, чем идеальный chunking без ответа в день 5. Это прямо по методичке: «Just get something working. Start with strong models & simple prompts.»

И ещё: **писать отчёт и собирать материал для питча — не в последний день, а параллельно каждый день** (15 минут вечером: что сделал, какие цифры, какой скриншот). Иначе раздел «Implementation» (50%) и питч (10%) будут писаться в панике.

## План по дням

### День 1 — Фундамент данных + GCP
Цель к вечеру: чанки с метаданными лежат в Chroma, поиск что-то возвращает.
- Настроить GCP-проект, включить Vertex AI, проверить квоту на Gemini и embeddings (если ещё нет — это первое, без этого всё стоит).
- Вытянуть законы с RIS (HTML→MD) — чище, чем парсить PDF. Кейсы docx→MD через pandoc.
- Проставить метаданные (`country, doc_type, law_code, paragraph, lang, source_url`).
- Структурный chunking по § + overlap. Глазами проверить ~10 чанков.
- Эмбеддинг (gemini-embedding-001, dim 768, правильные task_type) → Chroma.
- Smoke-тест: задать вопрос, посмотреть, релевантны ли найденные куски.

### День 2 — Первый сквозной ответ + демо-панель
Цель к вечеру: работает «вопрос → ответ со ссылкой на §», видно в Streamlit.
- LangGraph-граф: `retrieve → generate` (пока линейно, без guardrail).
- Системный промпт (grounding + цитата + дисклеймер) + 2-3 few-shots в стиле эталонов.
- Streamlit-панель: поле вопроса, ответ, показ найденных чанков и источников (это твоя «отладочная рентген-панель» — увидишь, что реально достаётся).
- Это уже демонстрируемый baseline.

### День 3 — Evaluation harness (это делает проект «научным»)
Цель к вечеру: таблица с метриками baseline на 13 вопросах.
- Скрипт: прогнать 13 вопросов через пайплайн, собрать ответы и найденные контексты.
- Метрики: retrieval (Hit Rate@k, MRR) + answer quality (RAGAS faithfulness/correctness ИЛИ свой Gemini-judge с рубрикой).
- Расширить тест-сет синтетикой до ~30-50 (отдельно от 13 человеческих).
- Зафиксировать baseline-числа. Это точка отсчёта.

### День 4 — Анализ + одно улучшение (история проекта)
Цель к вечеру: таблица «до/после» с приростом по метрике.
- Понять по метрикам: поиск виноват или генерация?
- Выбрать 1-2 рычага под слабое место: contextual retrieval / гибрид (dense+BM25) / reranker / HyDE / Flash-vs-Pro по цене.
- Прогнать eval снова, сделать таблицу до/после. Записать вывод словами.
- Это твой главный слайд и абзац в отчёте.

### День 5 — Guardrail + Router (безопасность и fallback)
Цель к вечеру: оффтоп/PII отсекаются, неуверенные ответы уходят к «человеку».
- Guardrail-узел: Google DLP (PII) + Gemini Flash (тема/инъекция) → структурный вердикт. Оригинал не сохраняется.
- Мини-eval guardrail на ~40 размеченных примерах → confusion matrix (вот тут Accuracy/Precision/Recall/F1 уместны).
- Router после генерации: Flash-judge уверенности → ниже порога → текст с контактом координатора + очищенный вопрос.

### День 6 — Telegram + GCP-прод (витрина)
Цель к вечеру: живой бот отвечает в Telegram.
- `python-telegram-bot`: локально polling. Подключить готовый LangGraph-граф.
- BigQuery: таблицы `interactions` и `guardrail_events` (без оригиналов). Писать туда события.
- Простой rate-limit (счётчик; для POC можно in-memory/Firestore).
- Деплой на Cloud Run (webhook). Проверить end-to-end из телефона.
- Если деплой упирается — оставь polling локально + видео-демо; это не стоит потерянного дня.

### День 7 — Отчёт, питч, подстраховка
- Дописать отчёт (структура = разделы задания: проблема, ресёрч, архитектура, имплементация, KPI).
- Слайды (8 мин): hook → проблема → решение → архитектура (твоя диаграмма) → evaluation (таблица до/после) → демо (видео!) → impact → next steps.
- **Снять видео демо обязательно** (методичка: «live demo? make sure you have a video as backup»).
- Финальная диаграмма архитектуры (отрисовать `01`-диаграмму начисто).

## Если время поджимает — порядок «что резать»
Режь с конца списка приоритетов, никогда не из Кольца A:
1. Сначала жертвуем Cloud Run-деплоем (оставляем локальный бот + видео).
2. Потом Telegram (оставляем Streamlit-демо).
3. Потом BigQuery (оставляем CSV-логи).
4. Router и Guardrail упрощаем до одного промпта.
**Ядро (RAG + eval + одно улучшение) не режется никогда.** Лучше идеальное ядро без Telegram, чем Telegram без измеренного RAG.

## Как эффективно работать с Claude Code (Opus 4.8) — практика

Ты будешь писать код моими руками. Чтобы это было быстро и без хаоса:

1. **Дроби задачи на узлы.** Не «сделай мне RAG», а «сделай модуль chunking: вход — папка MD, выход — список чанков с метаданными, разрезка по §, overlap 15%». Один узел = один понятный вход/выход. Так и код чище, и я реже ошибаюсь.
2. **Сначала тест/критерий, потом код.** Говори «вот 3 вопроса, ответ должен содержать § и не выдумывать» — я смогу проверять себя.
3. **Держи `CONTEXT.md` в репо** с текущим состоянием (что готово, какие числа eval, что дальше). После долгих сессий я смогу быстро восстановиться.
4. **Veni — фиксируй рабочие версии (git commit) часто.** Откатиться к рабочему состоянию важнее, чем красота.
5. **Осторожно с .venv на Desktop.** У тебя уже была повторная порча venv из-за синхронизации iCloud на Desktop. Держи venv ВНЕ синхронизируемой папки (например, `~/venvs/ambassy`) или используй контейнер. Это сэкономит тебе часы.
6. **Параллель моделей для разработки.** У тебя есть Claude Code, Codex, Gemini CLI. Codex/Gemini хороши для второго мнения на сложном баге или для генерации синтетических тест-вопросов пачкой. Но один «хозяин репозитория» (я) — чтобы не было конфликтов правок.
7. **Секреты — в переменные окружения / Secret Manager, не в код.** Telegram-токен, ключи GCP. И `.gitignore` на них (репо публичный на GitHub по заданию).
