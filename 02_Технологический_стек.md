## Разбор по узлам (выбор → почему → альтернативы)

### Узел 1. Оркестрация — LangGraph

**Почему.** Ты уже прошёл его в ДЗ-3, не тратишь время на новый фреймворк. LangGraph даёт три нужные вещи: (1) граф узлов = твоя схема один-в-один; (2) типизированное состояние между узлами = твоя «память состояния потока»; (3) checkpointer = сохранение состояния на каждом шаге, что закрывает требование «сохранять состояние на каждом узле» почти бесплатно.

**Альтернативы:**
- *Чистый Python-скрипт без фреймворка.* Плюс: ноль абстракций, проще для линейного пайплайна. Минус: сам пишешь состояние, ветвление (router), персист. Для линейного RAG — допустимо, но граф в питче выглядит профессиональнее.
- *Vertex AI Agent Engine / ADK (Google).* Плюс: «всё в Google», managed-деплой агентов. Минус: новый для тебя API за неделю = риск; меньше контроля; завязка на вендора. Не сейчас.
- *LlamaIndex.* Плюс: очень быстрый старт именно для RAG (готовые загрузчики, pipeline). Минус: ещё один фреймворк к изучению; LangGraph тебе ближе. Можно взять отдельные куски LlamaIndex (парсеры) внутри LangGraph.

**Вердикт:** LangGraph как каркас. Это твой «знакомый дом».

### Узел 2. Интерфейс — Telegram на Cloud Run

**Почему.** Цель — живой бот. `python-telegram-bot` — зрелая библиотека. Режим **webhook** (а не polling) разворачивается как HTTP-сервис на Cloud Run: serverless, масштабируется в ноль, платишь за вызовы — идеально для демо и «production mindset» в питче.

**Альтернатива по интерфейсу для подстраховки:** **Streamlit**. Плюс: поднимается за час, удобно показывать метрики/источники на экране, отлично для видео-демо. Минус: это не «настоящий канал» беженца. **Совет ментора:** сделай Streamlit СНАЧАЛА (как панель отладки RAG в Кольце A), а Telegram — поверх готового ядра в Кольце B. Если Telegram не успеешь — у тебя всё равно есть рабочее демо.

**Polling vs webhook:** для локальной разработки polling проще (не нужен публичный URL). Для прода на Cloud Run — webhook. Начни с polling локально, переключись на webhook при деплое.

### Узел 3. Destructive checks — rate-limit + дешёвый предфильтр

**Почему (две разные задачи под одним названием):**
1. *Защита от потока сообщений* — это инфраструктура. Счётчик «N сообщений за T секунд по `telegram_id`». Хранилище: in-memory dict для POC; BigQuery/Firestore/Redis для прода. LLM тут не нужен.
2. *Защита бюджета от не-по-теме запросов* — дешёвый классификатор `Gemini 3.1 Flash-Lite` ДО дорогого Pro. Отсекает «напиши код / нарисуй картинку» за доли цента.

**Альтернативы для rate-limit:**
- *Firestore* — плюс: realtime, простые атомарные счётчики, serverless. Минус: ещё один сервис.
- *Redis (Memorystore)* — плюс: классика rate-limit, быстро. Минус: стоит денег даже в простое, overkill для POC.
- *BigQuery* — плюс: уже в стеке, всё в одном месте для аналитики. Минус: BQ не для частых мелких записей/чтений в реальном времени (он аналитический, не транзакционный). **Нюанс:** для счётчиков лучше Firestore, а BigQuery — для пост-аналитики событий. Об этом честно скажи в trade-offs.

**Эскалирующие баны:** опиши в архитектуре (таблица состояний юзера, счётчик банов, растущая длительность). Не строй за неделю — это бизнес-логика без GenAI-ценности для оценки.

### Узел 4. Guardrails — DLP (PII) + Gemini Flash (тема/инъекция)

Это два под-чека. Раздели их — разные инструменты.

**4a. PII-редактирование — Google Cloud DLP API.**
Почему: DLP — специализированный сервис Google для поиска и маскирования персональных данных (имена, телефоны, email, паспорта, адреса) на 100+ типов, с поддержкой многих языков. Это «правильный инструмент для правильной задачи». Бытовая аналогия: чтобы найти все паспорта в стопке бумаг, нанимаешь обученного клерка (DLP), а не просишь поэта (LLM) «посмотри, нет ли тут личного».
- *Альтернатива — regex.* Плюс: бесплатно, быстро. Минус: ломается на вариативности (телефоны в разных форматах, имена). Только как грубая сетка.
- *Альтернатива — LLM (Gemini) для PII.* Плюс: понимает контекст («меня зовут…»). Минус: можно подвергнуть инъекции, недетерминирован, дороже. **Лучшее решение:** DLP как основной слой + LLM как дополнительный для контекстных случаев.

**4b. Проверка темы и prompt-injection — Gemini 3.5 Flash.**
Один вызов возвращает структурированный вердикт: `{on_topic: bool, category: str, injection_suspected: bool}`. Flash дёшев и мультиязычен (поймёт и русский, и белорусский).
- *Альтернатива — Vertex AI safety filters / Model Armor.* Google даёт встроенные фильтры безопасности и Model Armor для защиты от инъекций/jailbreak. Плюс: managed, не надо писать промпт. Минус: фильтрует «вредность», но не «по теме ли это про иммиграцию» — тематический чек всё равно твой. **Совет:** включи встроенные safety-фильтры Vertex как нижний слой + свой Flash-классификатор темы сверху.
- *Альтернатива — отдельная guard-модель (напр. ShieldGemma).* Плюс: заточена под безопасность. Минус: ещё одна модель в инфраструктуре. Для POC хватает Flash + встроенных фильтров.

> Архитектурное правило (повтори в отчёте): оригинал сообщения НИКУДА не пишется. Guardrail на выходе отдаёт только очищенный текст; сырой запрос живёт в оперативной памяти запроса и умирает с ним. В лог/BigQuery идут только: факт события, категория, очищенный текст, timestamp, `telegram_id`-хэш.

### Узел 5. Query enrich — Gemini 3.5 Flash

**Почему.** Один дешёвый вызов делает cross-lingual мост: переписывает русский/белорусский вопрос в (а) нормализованную формулировку, (б) перевод/ключевые термины на немецком (чтобы попасть в немецкие законы), (в) расширение синонимами. Это прямо повышает recall retrieval при разрыве языков.

**Альтернативы:**
- *HyDE (Hypothetical Document Embeddings).* Модель генерирует «гипотетический идеальный ответ», и ищем по нему. Плюс: часто сильно поднимает retrieval. Минус: +1 генерация (латентность/стоимость). Отличный кандидат на «improve one dimension».
- *Multi-query expansion.* Генерируешь 3-4 переформулировки, ищешь по всем, объединяешь. Плюс: ловит разные формулировки. Минус: дороже, дедупликация. Тоже рычаг улучшения.
- *Без enrich, полагаться на мультиязычный embedding.* Плюс: проще, меньше латентность. Минус: на разрыве RU↔DE recall просядет. **Совет:** сделай baseline БЕЗ enrich (как велит методичка — простейшее), измерь, потом добавь enrich/HyDE и покажи прирост. Это и есть твоя история улучшения.

### Узел 6. Embeddings — gemini-embedding-001

**Почему.** На июнь 2026 это топ MTEB multilingual, 100+ языков (русский уверенно, белорусский — частично), вход до 2048 токенов, размерность 3072 с Matryoshka (можно урезать до 1536/768 без заметной потери — экономия памяти/скорости поиска). Для cross-lingual RU↔DE это сильный дефолт, и он «родной» для Vertex.

**Альтернативы:**
- *text-multilingual-embedding-002 (Vertex, старее).* Плюс: дешевле, проверенный. Минус: слабее на бенчмарках. Фолбэк.
- *gemini-embedding-2 (мультимодальная).* Плюс: текст+картинки+аудио. Минус: тебе не нужна мультимодальность — лишняя сложность/цена.
- *Open-source BGE-M3 / multilingual-e5-large.* Плюс: бесплатно, можно self-host, BGE-M3 силён в cross-lingual и даёт сразу dense+sparse. Минус: надо где-то крутить (GPU/CPU), отходишь от «Google stack». Хороший аргумент для слайда «как удешевить» (убрать платные эмбеддинги).

> Практическая деталь: эмбедь не «голый чанк», а «чанк + его контекстное описание + метаданные» (contextual retrieval). И помни лимит 2048 токенов на вход — твои чанки должны быть меньше. Детали в `03`.

### Узел 7. Vector store — Chroma (POC) → BigQuery/Vertex (прод)

**POC: ChromaDB (или FAISS).** Почему: ноль инфраструктуры, живёт в файле/памяти, мгновенные итерации над chunking/embeddings. На корпусе из нескольких законов это летает. Аналогия: пока пробуешь рецепт — готовишь на своей кухне, а не строишь ресторан.

**Прод, вариант 1: BigQuery Vector Search.** Плюс: у тебя там уже abuse-DB и long-term memory — ОДНА платформа данных, единый SQL, дешёвое хранение, аналитика метрик в том же месте. Минус: задержки поиска выше, чем у специализированного индекса; не для тысяч QPS. Для твоих объёмов — отлично и связно в питче.

**Прод, вариант 2: Vertex AI Vector Search.** Плюс: managed ANN-индекс, низкая латентность, масштаб. Минус: дороже, сложнее настройка (index/endpoint/deploy), избыточно для малого корпуса.

**Альтернатива: pgvector на Cloud SQL/AlloyDB.** Плюс: привычный Postgres, фильтрация по метаданным + вектор в одном запросе, гибрид удобно. Минус: managing БД. Хороший средний путь, если любишь SQL.

**Вердикт:** Chroma для разработки, BigQuery Vector Search для прод-демо (связность стека > пик производительности на учебном объёме).

### Узел 8. Retrieval + Reranker

**Retrieval.** Двухступенчато (как ты и задумал): (1) жёсткий фильтр по метаданным `country` (и при желании `doc_type`), (2) гибридный поиск внутри: dense (эмбеддинги, смысл) + sparse (BM25/ключевые слова, точные термины и номера §). Юридические запросы часто содержат точные маркеры («§88», «NAG») — чистый смысловой поиск их теряет, поэтому гибрид.

**Reranker — Vertex AI Ranking API.** После гибрида берёшь топ-20 кандидатов и переранжируешь моделью-ранкером в топ-5. Это дешёвый и сильный рычаг precision. Аналогия: сначала грубое сито (быстро достать 20 правдоподобных), потом эксперт-дегустатор (ранкер) выбирает 5 лучших.
- *Альтернатива — Cohere Rerank multilingual.* Плюс: очень силён, мультиязычный. Минус: вне Google-стека.
- *Альтернатива — без ранкера.* Проще/быстрее, но precision ниже. **Совет:** baseline без ранкера → добавить ранкер → показать прирост (опять «improve one dimension»).

### Узел 9. LLM «мозг» — Gemini 3.5 Pro → Flash

**Почему именно так.** Методичка велит: «start with the strongest model». Сильнейший в линейке — `Gemini 3.5 Pro` (флагман; если по квоте недоступен — `gemini-3.1-pro-preview`). Делаешь baseline на Pro, фиксируешь качество. Затем пробуешь `Gemini 3.5 Flash`: если качество просело незначительно, а цена/латентность упали в разы — это твоя готовая история «improve cost dimension».

**Альтернативы (для слайда сравнения):**
- *Claude / GPT* — часто сильнее в рассуждении и следовании инструкциям. Минус: вне Google-стека, отдельный биллинг. Можно упомянуть как «рассматривали, выбрали Gemini ради единой экосистемы и кредитов» — это валидный trade-off.
- *Gemini 3.1 Flash-Lite* — для самых дешёвых под-узлов (guardrail/router/enrich), не для финальной генерации.

> Для генерации обязательно: (1) системный промпт с ролью «юридический ассистент по иммиграции в страну X», (2) few-shots в стиле твоих эталонных ответов, (3) требование цитировать § и источник, (4) дисклеймер «не заменяет юриста», (5) инструкция «если в контексте нет ответа — честно скажи, не выдумывай» (анти-галлюцинация).

### Узел 10. Router после LLM — Gemini Flash как judge

**Почему.** После генерации второй дешёвый вызов оценивает ответ: есть ли он в источниках (grounding), уверенный ли, полный ли. Ниже порога → отдаём контакт координатора + очищенный вопрос ему. Это закрывает «fallback for unanswered questions» и снимает юридический риск.

**Альтернативы:**
- *Confidence по logprobs/самооценке генератора.* Плюс: без лишнего вызова. Минус: самооценка LLM ненадёжна. Judge отдельной моделью честнее.
- *RAGAS faithfulness как порог.* Плюс: формальная метрика. Минус: считается оффлайн, не в реальном времени. Используй RAGAS для отчётной оценки, Flash-judge — для рантайм-роутинга.

---

### Узел 11. Хранилища — BigQuery (Guardrail DB + Long-term)

**Почему BigQuery.** Уже в твоём стеке; serverless; SQL-аналитика «из коробки» — а тебе для отчёта нужны как раз агрегаты (сколько запросов, сколько ушло к человеку, топ-тем, сколько guardrail-срабатываний). Один SELECT — и у тебя цифры для слайдов.

**Схемы (минимум):**
- `guardrail_events`: `event_id, telegram_id_hash, category, sanitized_text, ts, action_taken`.
- `interactions`: `interaction_id, telegram_id_hash, sanitized_query, country, retrieved_sources[], answer, judge_score, routed_to_human(bool), ts`. **Без оригинала сообщения.**

**Нюанс (повтори в trade-offs):** BigQuery — аналитическое хранилище, не транзакционное. Для частых realtime-счётчиков (rate-limit, баны) лучше Firestore; BigQuery — для журналирования и аналитики. Не делай rate-limit на BigQuery.

**Альтернатива — Firestore для всего оперативного** (состояния юзера, баны, счётчики) + BigQuery для аналитики. Это «правильная» прод-архитектура: горячие данные в Firestore, холодные в BQ. Можешь так и описать в архитектуре, реализовать по минимуму.
