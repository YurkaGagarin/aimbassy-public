## 1. Вердикт в двух абзацах (verdict-first)

**Идея — сильная, социально значимая и хорошо ложится на критерии задания.** У тебя есть редкая для учебного проекта вещь: настоящая проблема, настоящие данные (австрийские законы + кейсы) и готовый эталонный тест-сет с ответами. Это золото для раздела Evaluation, который преподаватели ценят выше всего («it works» без цифр не считается).

**Но твоя архитектура из PRD рассчитана на команду из 3-4 человек на месяц, а не на соло за неделю.** Это не критика — это нормально на стадии мечты. Моя главная задача как ментора: помочь тебе отделить «оцениваемое ядро» (то, что приносит баллы и должно работать идеально) от «витрины архитектуры» (то, что ты опишешь и нарисуешь, но реализуешь по минимуму или отложишь). Если попытаешься сделать всё — не успеешь ничего довести до конца, и пострадает именно тот блок, который весит 50%.

## 2. Как устроена оценка (это диктует всё остальное)

Из файла задания, проценты от итоговой оценки:

| Раздел | Вес | Что реально проверяют |
|---|---|---|
| 1. Постановка проблемы | 5% | Чёткое, измеримое описание боли. У тебя уже есть. |
| 2. Маркет-ресёрч и technical discovery | 15% | Конкуренты, gap, аудитория, доступность данных, feasibility. |
| 3. Архитектура GenAI-системы | 20% | Диаграмма + таблица стека с обоснованием каждого выбора. |
| **4. Имплементация** | **50%** | **Работающий POC + честная evaluation + улучшение хотя бы одного измерения.** |
| 5. Питч классу | 10% | 8 минут + 4 мин Q&A. История, демо, цифры. |

Ключевые цитаты из методички (Project Breakdown), которые я буду повторять как мантру:

- «Start with the strongest model and the simplest possible prompt. Only optimize after you have a baseline.»
- «Improve **at least one** dimension (latency / cost / quality / robustness). You don't need to improve everything.»
- «Simplicity is fine: Clear, working > complex and fragile.»
- «A well-documented failure is worth more than a fudged success.»

Перевожу на человеческий: **тебе не нужна навороченная система. Тебе нужно одно работающее ядро (RAG-ответ на юридический вопрос), измеренное по метрикам, и одно осмысленное улучшение.** Всё остальное — бонус и материал для слайда «next steps».

## 3. Главная рекомендация по объёму (прочитай дважды)

Ты выбрал цель «Telegram + полный GCP-прод». Я её принимаю, но переупаковываю в безопасную последовательность. Принцип: **сначала то, что оценивается, потом то, что красиво.**

Три кольца приоритета:

**Кольцо A — ОЦЕНИВАЕМОЕ ЯДРО (must, ~60% времени).** Без него нет проекта.
- RAG-пайплайн по Австрии: подготовка файлов → chunking → embeddings → retrieval → генерация ответа на Gemini.
- Evaluation на твоих 13 эталонных вопросах (+ расширить синтетикой) с понятными метриками.
- Одно задокументированное улучшение (например, Flash vs Pro по цене/качеству, или +reranker к retrieval).
- Тонкий интерфейс для демо (Telegram-бот ИЛИ Streamlit) — чтобы было что показать вживую.

**Кольцо B — ВИТРИНА ПРОДА (should, ~25% времени).** Поднимает оценку за архитектуру и «production mindset».
- Деплой на Cloud Run (бот как webhook-сервис).
- Guardrails как один узел: PII-редактирование (Google DLP API) + проверка темы (Gemini Flash).
- Router после LLM: LLM-judge проверяет уверенность ответа, иначе отдаёт контакт координатора.
- Хранение в BigQuery (long-term + guardrail-события).

**Кольцо C — АРХИТЕКТУРНАЯ МЕЧТА (could / описать, но НЕ кодить за неделю, ~15% времени на описание).**
- DDOS-защита и система эскалирующих банов (это инфраструктура, а не GenAI — опиши, не строй).
- Интерактивный цикл уточняющих вопросов с сохранением состояния.
- Полноценный ReAct-агент с веб-поиском.
- Двухстрановость (Литва) — пока данных нет, держи как «дизайн на масштабирование».

> Если в конце недели Кольцо A работает и измерено — у тебя уже крепкая оценка. Каждый элемент B — плюс. C — это слайд «дайте две недели и бюджет».

## 4. Самый важный технический поворот, который ты должен понять

**Ты перепутал два разных типа метрик.** В PRD ты пишешь, что будешь мерить качество RAG через Confusion Matrix (Accuracy / Precision / Recall / F1). Это частая и понятная ошибка. Разберём на бытовом примере.

- **Confusion Matrix** работает там, где ответ — «да/нет» или «класс A/B/C». Пример из жизни: тест на covid — он либо положительный, либо отрицательный, и мы считаем, сколько раз тест ошибся. Это про **классификаторы**.
- **Качество ответа RAG** — это свободный текст («Около четверти получают убежище, статистика падает...»). Тут нет «правильного класса». Нельзя сказать «этот абзац — false positive».

Поэтому метрики распределяются так (подробно в `03_RAG_дизайн.md`):

- Confusion Matrix (Accuracy/Precision/Recall/F1) — правильно применять к **Guardrail-классификатору** («запрос по теме / не по теме», «есть PII / нет PII») и к **retrieval** («нашли ли мы нужный документ среди топ-k»).
- Качество ответа RAG — мерить через **LLM-as-judge** (RAGAS-подход): *faithfulness* (ответ не выдуман, опирается на источник), *answer correctness* (совпадает по смыслу с эталоном), *context precision/recall* (релевантность найденных кусков).

Это не придирка — это прямо влияет на то, как ты построишь таблицу evaluation в отчёте. И это отличная тема для вопроса учителю (см. `06`).
